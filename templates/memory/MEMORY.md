## Feedback

Six foundational feedback rules ship as seeds — these apply to any project. Keep them unless you have a deliberate reason to remove one; add project-specific feedback alongside.

- [Never assume](feedback_never_assume.md) — verify before claiming; single grep doesn't prove a thing doesn't exist
- [Trust but verify](feedback_trust_but_verify.md) — questions deserve fresh evidence, every time
- [Completeness audit](feedback_completeness_audit.md) — structured self-audit before declaring done
- [Constructive review](feedback_constructive_review.md) — proactively flag business / security / UX / observability / architectural concerns on every feature
- [Explainability gate](feedback_explainability_gate.md) — READMEs as code-quality signal; hard-to-write README = refactor signal
- [Never override CI](feedback_never_override_ci.md) — no `--admin`, `--force`, or `--no-verify`, no exceptions
- [Example feedback](feedback_example.md) — format reference; delete or rename as your own feedback accumulates
